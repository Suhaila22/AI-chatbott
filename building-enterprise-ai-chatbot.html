Under the Hood: Transformer Architecture & Attention

You cannot tune what you do not understand. Every modern enterprise LLM — LLaMA, Mistral, Falcon, GPT variants — is a transformer. The transformer is not magic; it is four stacked components working in concert:

The Four Core Components

  • Embedding Layer — converts discrete tokens into dense continuous vectors of dimensionality d (typically 4096 for a 7B model). This is where language becomes geometry.
  • Positional Encoding — injects sequence-order information into otherwise order-agnostic embeddings. Without this, "dog bites man" and "man bites dog" would be identical to the model.
  • Self-Attention (Multi-Head) — the computational heart. Each token generates three vectors: Query (Q), Key (K), and Value (V). Attention scores are computed as softmax(QKT / √dk) and used to weight the value vectors. This is what gives transformers their ability to capture long-range dependencies that destroyed earlier RNN architectures.
  • Feed-Forward Network — a position-wise MLP applied after attention, typically expanding to 4× the model dimension before projecting back. This is where the model's "knowledge" is primarily stored.
Python # Self-attention — the mathematical core import torch import torch.nn.functional as F def scaled_dot_product_attention(Q, K, V, mask=None): d_k = Q.size(-1) # Score: how much each query attends to each key scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) weights = F.softmax(scores, dim=-1) return torch.matmul(weights, V), weights

Multi-head attention runs this mechanism h times in parallel, each head learning to attend to different aspects of the input — syntactic structure, semantic similarity, coreference, factual associations. The outputs are concatenated and projected. This parallel structure is why transformers scale so dramatically with compute: there is no sequential dependency across heads.

Understanding self-attention is critical because LoRA — the dominant fine-tuning technique — specifically targets the query and value projection matrices of the attention mechanism. If you don't know why those matrices matter, you will tune the wrong things.

Model Size vs. Hardware Reality

The rule of thumb is ~2 bytes per parameter for float16 inference. A 7B model needs ~14 GB VRAM at minimum, 13B needs ~26 GB, and a 70B model requires multi-GPU configurations or aggressive quantisation. In practice:

  • 7B models — consumer RTX 3090/4090 (24 GB). Sufficient for most business chatbot tasks.
  • 13B models — enterprise single GPU (A100 40 GB) or consumer GPU + QLoRA quantisation.
  • 70B models — A100 80 GB minimum, preferably multi-GPU. Necessary for complex reasoning, legal analysis, medical summarisation.

Data Engineering for Fine-Tuning

The model is only as good as the data you feed it. Enterprise fine-tuning data lives in four places: HR and policy documents (SharePoint, Confluence), customer support logs and CRM tickets, product manuals and technical documentation, and structured databases exposed via APIs. The pipeline from raw source to training-ready dataset is non-trivial.

Tokenisation: The Critical Detail Engineers Miss

You must use the exact tokeniser that was used to pre-train your base model. LLaMA uses a SentencePiece-based BPE tokeniser; Mistral uses the same. BERT uses WordPiece. Mixing tokenisers — even between models that look similar — is a silent catastrophe that produces garbage training gradients without throwing an error.

Tokeniser mismatch is one of the most common silent failures in enterprise fine-tuning projects. The training loop will complete without errors, evaluation loss will look reasonable, and the model will hallucinate confidently on every deployment query. Always verify: tokenizer.name_or_path must match your base model exactly.

Splitting, Validation, and Compliance

Use an 80/10/10 train/validation/test split with strict non-overlap enforcement. Data leakage — where validation or test examples contaminate the training set — is one of the most common causes of overfit models that fail in production. Hash every example; deduplicate across splits before training begins.

On the compliance side: all PII must be anonymised or removed before data enters any training pipeline. Depending on your jurisdiction, GDPR Article 17 ("right to erasure") may require you to delete a user's data from a trained model — which is technically non-trivial. The pragmatic solution is aggressive pre-training anonymisation so that no PII is ever ingested.

PEFT: LoRA & QLoRA — Fine-Tuning Without a Supercomputer

Full fine-tuning of a 7B model updates all 7 billion parameters, requires ~70 GB of VRAM, and costs $5,000–$10,000 per run on A100 hardware. This puts fine-tuning out of reach for most organisations. Parameter-Efficient Fine-Tuning (PEFT) changes the economics radically.

LoRA: Low-Rank Adaptation

LoRA operates on the insight that weight updates during fine-tuning have a low intrinsic rank — meaning they live in a much smaller subspace than the full weight matrix. Instead of computing ΔW directly, LoRA decomposes it:

Math → Code # LoRA core formula: W_new = W_0 + ΔW = W_0 + B·A # W_0: frozen pre-trained weights (4096×4096 = 16.7M params) # B: trainable (4096×r) # A: trainable (r×4096) # With r=8: (4096×8) + (8×4096) = 65,536 params ← 99.6% reduction from peft import LoraConfig, get_peft_model, TaskType lora_config = LoraConfig( task_type=TaskType.CAUSAL_LM, r=8, # rank — lower = more efficient, less expressive lora_alpha=16, # scaling: effective scale = alpha/r = 2.0 lora_dropout=0.1, # regularisation for small datasets target_modules=["q_proj", "v_proj"], # query & value attention matrices bias="none" ) model = get_peft_model(base_model, lora_config) model.print_trainable_parameters() # → trainable params: 4,194,304 || all params: 6,738,415,616 # trainable %: 0.0623

QLoRA: When You Only Have a Consumer GPU

QLoRA adds 4-bit NormalFloat (NF4) quantisation of the frozen base weights on top of LoRA. This reduces a 65B model from ~130 GB to ~48 GB — making it feasible on a single A100. For 7B models, QLoRA runs on an RTX 3090 with 24 GB VRAM.

For most enterprise chatbot use cases, QLoRA delivers a 93–97% performance ceiling at 3–5% of the full fine-tuning cost. The remaining 3–7% performance gap is rarely observable in production — users cannot distinguish a model that achieves 96% vs 100% on academic benchmarks when answering real HR policy questions.

RAG Pipelines: Grounding Your Chatbot in Real Knowledge

Fine-tuning teaches the model how to answer in your domain. Retrieval-Augmented Generation (RAG) gives it access to what to answer from — your live knowledge base, updated daily, without retraining. This is the architecture that separates enterprise chatbots from toy demos.

The Two-Phase RAG Architecture

RAG operates in two phases: an offline ingestion phase that processes your document corpus into a vector database, and an online query phase that retrieves relevant chunks at runtime before passing them to the LLM.

Python · LangChain RAG Pipeline from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain.embeddings import OpenAIEmbeddings from langchain.vectorstores import Pinecone from langchain.chains import RetrievalQA # ── Offline: Ingest Documents ────────────────────────────── splitter = RecursiveCharacterTextSplitter( chunk_size=512, # tokens per chunk chunk_overlap=50, # overlap prevents context loss at boundaries separators=["\n\n", "\n", " "] ) chunks = splitter.split_documents(raw_docs) embeddings = OpenAIEmbeddings(model="text-embedding-ada-002") vectorstore = Pinecone.from_documents(chunks, embeddings, index_name="hr-knowledge-base") # ── Online: Query Pipeline ───────────────────────────────── retriever = vectorstore.as_retriever( search_type="similarity", search_kwargs={"k": 5} # top-5 most relevant chunks ) qa_chain = RetrievalQA.from_chain_type( llm=local_llm, # your fine-tuned Ollama model retriever=retriever, return_source_documents=True ) response = qa_chain({"query": "What is our work-from-home policy?"})

Why RAG Beats Knowledge-Embedded Fine-Tuning

  • Recency — Policy changes, product updates, and org restructuring are reflected in the knowledge base on the next ingestion cycle. No retraining required.
  • Auditability — Every response can be traced to specific source documents. The model cites its sources; compliance teams can audit the chain of reasoning.
  • Reduced Hallucination — An LLM grounded in retrieved, verified documents is significantly less likely to confabulate plausible-sounding but incorrect answers.
  • Scalability — Your knowledge base can grow to millions of documents without retraining. The embedding index scales horizontally; the model does not.

A chatbot that exposes employee payroll data to an intern because someone forgot to scope the API context is not an AI failure — it is an engineering failure. Security cannot be bolted on post-deployment; it must be architected from the first design review.

The Continuous Retraining Imperative

Static models decay. Data drift — statistical shifts in user query patterns — and concept drift — changes in the underlying relationships the model predicts — are both inevitable. An HR chatbot trained on 2024 policies will confidently give wrong answers after 2025 policy updates unless it is retrained.

Build automated monitoring that triggers retraining when: (a) validation accuracy drops below threshold, (b) embedding distribution of incoming queries drifts significantly from training distribution (measured by KL divergence or MMD), or (c) a scheduled calendar interval passes. Use A/B shadow deployment to validate retrained models before full rollout.

Common Pitfalls Engineers Fall Into

Catastrophic Forgetting. Fine-tuning a narrow domain dataset can erase the model's broad pre-trained capabilities. PEFT (LoRA/QLoRA) mitigates this significantly because the base weights remain frozen. If you must full-fine-tune, mix general-purpose data at an 80/20 ratio with your domain data and keep the learning rate below 5e-5.

Overfitting to Training Examples. Classic signs: training accuracy 99%, validation accuracy 62%. Fixes: increase dataset size (target >1,000 high-quality examples), add dropout to LoRA layers (lora_dropout=0.1), use weight decay (0.01), implement early stopping on validation loss. Do not just add more epochs — that is the wrong direction.

Ignoring Retrieval Quality Before LLM Integration. Engineers wire up the full RAG pipeline and then wonder why answers are wrong. Evaluate your retriever in isolation first: given a test query, do the top-5 retrieved chunks contain the correct information? Precision@5 should exceed 80% before you connect the LLM. Bad retrieval + good LLM = confident hallucination.

Deploying Without Resource Limits. A single poorly optimised query can OOM a GPU container and crash inference for all concurrent users. Always set --memory and --cpus flags in Docker; set Kubernetes resource limits/requests; implement request queuing and circuit breakers. Test under load before production.

Security as an Afterthought. Prompt injection — where a user's input manipulates the model into revealing system prompt content or executing unauthorised actions — is a real attack vector. Implement input sanitisation, output filtering, and context isolation at the application layer. Your LLM's safety training is not a security control; it is a probabilistic behaviour.

Where to Go From Here

This post has covered the conceptual and technical depth required to architect and build an enterprise AI chatbot from the ground up. The natural next steps for any team taking this seriously:

  1. Stand up a local Ollama instance and benchmark a 7B model against your use case before committing to hardware.
  2. Run a data audit — map every internal knowledge source and estimate the volume of clean, structured training examples you can extract.
  3. Run a LoRA fine-tuning experiment on a 1,000-example pilot dataset. Validate with domain experts before scaling.
  4. Build the RAG pipeline in isolation. Evaluate retrieval quality independently. Integrate the LLM only when retrieval is solid.
  5. Containerise, monitor, and load test before any end-user exposure. Define your rollback procedure before you deploy.

Building reliable AI systems is hard engineering work. The organisations that do it well — with rigorous data pipelines, principled fine-tuning, robust retrieval, and serious security — will have systems that compound in value over time. The ones that treat it as a prompt-engineering project will have expensive technical debt and user trust deficits that are very difficult to recover from.

Build it right the first time.
